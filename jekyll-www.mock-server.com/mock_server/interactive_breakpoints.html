Interactive breakpoints let you pause proxied or forwarded HTTP exchanges, inspect the request or response, optionally modify it, and then continue or abort. This is useful for debugging integrations, testing error handling with modified responses, and inspecting streaming traffic (SSE, chunked, gRPC, WebSocket, GraphQL subscription) frame by frame.

Breakpoints are off by default and have zero overhead when disabled.

Breakpoint Phases

You can pause an exchange at four points:

PhaseWhat is pausedActions available
Request The outbound request, before it is forwarded to the upstream server Continue (forward as-is), Modify (forward a changed request), Abort (return error to client)
Response The upstream response, before it is written to the client Continue (write as-is), Modify (write a changed response), Abort (return error to client)
Stream frame Each individual frame of a streaming response (SSE, HTTP/1.1 chunked, gRPC server-streaming, WebSocket, or GraphQL subscription) Continue, Modify, Drop (discard frame), Inject (write original + extra frame), Close (end stream)
Inbound frame Each client-to-server frame on a bidirectional connection (WebSocket or GraphQL subscription). Experimental, off by default. Continue, Modify, Drop (discard frame), Inject (process original + extra frame), Close (close connection)
 

Enabling Breakpoints

Each phase has its own configuration property. Enable only the phases you need:

PropertyEnv varDefaultDescription
mockserver.breakpointEnabled MOCKSERVER_BREAKPOINT_ENABLED false Enable request-phase breakpoints
mockserver.breakpointResponseEnabled MOCKSERVER_BREAKPOINT_RESPONSE_ENABLED false Enable response-phase breakpoints
mockserver.breakpointStreamEnabled MOCKSERVER_BREAKPOINT_STREAM_ENABLED false Enable stream-frame breakpoints (SSE, HTTP/1.1 chunked, gRPC server-streaming, WebSocket, GraphQL subscription)
mockserver.breakpointInboundEnabled MOCKSERVER_BREAKPOINT_INBOUND_ENABLED false Enable inbound frame breakpoints — intercept client-to-server frames on WebSocket and GraphQL subscription connections (experimental)
mockserver.breakpointTimeoutMillis MOCKSERVER_BREAKPOINT_TIMEOUT_MILLIS 30000 Auto-continue timeout in milliseconds (shared across all phases)
mockserver.breakpointMaxHeld MOCKSERVER_BREAKPOINT_MAX_HELD 50 Maximum number of concurrently paused exchanges or frames
 

Workflow

The typical workflow is:

  1. Enable the breakpoint phase(s) you want via configuration properties or the REST API
  2. Send a request through MockServer (proxy or forward mode)
  3. List paused exchanges — GET /mockserver/breakpoint returns all currently held exchanges with their IDs, phase, and captured request/response data
  4. Inspect the paused request or response
  5. Decide — continue, modify, or abort via the REST API (or click in the dashboard)

If you do not resolve a paused exchange within the timeout (default 30 seconds), it auto-continues.

 

Request-Phase Breakpoints

When breakpointEnabled is true, every proxied or forwarded request pauses before being sent to the upstream server. You can inspect the request and then:

  • Continue — forward the original request unchanged
  • Modify — forward a replacement request (e.g. add a header, change the body)
  • Abort — return an error response to the client without forwarding

Example: modify a paused request to add an authorization header:

# List paused exchanges
curl -s http://localhost:1080/mockserver/breakpoint

# Modify the request
curl -X PUT http://localhost:1080/mockserver/breakpoint/modify \
  -d '{
    "id": "pause-id-from-list",
    "httpRequest": {
      "headers": {
        "Authorization": ["Bearer test-token"]
      }
    }
  }'
 

Response-Phase Breakpoints

When breakpointResponseEnabled is true, the upstream response is held before being written to the client. This applies to non-streaming (buffered) responses only — streaming responses use stream-frame breakpoints instead.

The paused exchange includes the response statusCode and reasonPhrase in the list output. To modify the response: 

curl -X PUT http://localhost:1080/mockserver/breakpoint/modify \
  -d '{
    "id": "pause-id-from-list",
    "httpResponse": {
      "statusCode": 200,
      "body": "{\"modified\": true}"
    }
  }'
 

Stream-Frame Breakpoints

When breakpointStreamEnabled is true, each frame of a streaming response is held individually before being written to the client. This covers SSE, HTTP/1.1 chunked, gRPC server-streaming, WebSocket (eager/scripted and bidirectional), and GraphQL subscription pushes. It lets you inspect and manipulate streaming traffic frame by frame.

Available actions for each frame:

ActionEffect
ContinueWrite the original frame to the client
ModifyWrite a replacement body for this frame
DropDiscard the frame (client never sees it)
InjectWrite the original frame plus an extra frame after it
CloseEnd the stream at this frame (sends a final close and evicts remaining held frames)

Frames within a stream must be resolved in order — resolving frame 3 before frame 2 is rejected. For SSE/HTTP-chunked streams and sequenced gRPC/WebSocket streams, the upstream is automatically back-pressured while a frame is held (the next chunk or message is not requested until the current frame is resolved). However, bidirectional WebSocket and GraphQL subscription pushes are not back-pressured — frames are parked concurrently as they arrive and can hit the breakpointMaxHeld cap under high throughput.

 

Inbound Frame Breakpoints

When breakpointInboundEnabled is true, each client-to-server frame on a bidirectional WebSocket or GraphQL subscription connection is held before MockServer processes it. This lets you inspect, modify, drop, or inject inbound frames — useful for testing how an upstream server handles manipulated client messages.

Inbound frame breakpoints cover WebSocket bidirectional, GraphQL subscription, and gRPC bidirectional connections. For gRPC-bidi, the implementation releases each HTTP/2 DATA frame immediately (refunding the flow-control window) and uses pull-based ctx.read() backpressure. This feature is off by default and experimental.

Inbound frames use the same REST API endpoints as stream-frame breakpoints (/mockserver/breakpoint/stream/*). The /mockserver/breakpoint/streams list endpoint includes a direction field (INBOUND for client-to-server, OUTBOUND for server-to-client) so you can distinguish them.

 

REST API Reference

Request / response breakpoints

MethodPathDescription
GET/PUT/mockserver/breakpointList all paused exchanges (includes phase, request/response data)
PUT/mockserver/breakpoint/continueContinue a paused exchange: {"id": "..."}
PUT/mockserver/breakpoint/modifyModify: {"id": "...", "httpRequest": {...}} for request phase, or {"id": "...", "httpResponse": {...}} for response phase
PUT/mockserver/breakpoint/abortAbort: return error to client without forwarding: {"id": "..."}

Stream-frame breakpoints

MethodPathDescription
GET/mockserver/breakpoint/streamsList all held frames grouped by stream
PUT/mockserver/breakpoint/stream/continueContinue a held frame: {"id": "..."}
PUT/mockserver/breakpoint/stream/modifyModify a frame: {"id": "...", "body": "..."}
PUT/mockserver/breakpoint/stream/dropDrop a frame: {"id": "..."}
PUT/mockserver/breakpoint/stream/injectInject after a frame: {"id": "...", "body": "..."}
PUT/mockserver/breakpoint/stream/closeClose stream at a frame: {"id": "..."}
 

Dashboard UI

The MockServer dashboard includes a Breakpoints panel that shows all currently paused exchanges. Each paused exchange displays:

  • A Phase indicator (REQUEST or RESPONSE)
  • For request-phase: the HTTP method and path
  • For response-phase: the status code and reason phrase
  • Continue, Modify, and Abort buttons

The Modify dialog provides a JSON editor: edit the request JSON for request-phase exchanges, or the response JSON for response-phase exchanges. Click "Apply" to send the modified exchange.

 

Safety Rails

  • Timeout auto-continue: paused exchanges auto-continue after breakpointTimeoutMillis (default 30 seconds) if not resolved, preventing requests from hanging indefinitely.
  • Max-held cap: when breakpointMaxHeld (default 50) exchanges are held simultaneously, new intercepts are skipped (pass through without pausing).
  • Non-blocking: the breakpoint mechanism is fully asynchronous. No Netty event-loop or scheduler thread is blocked while waiting for a decision.
  • Stream-close eviction: when a stream completes or errors, all held frames for that stream are auto-continued.

Java Client

MockServerClient provides typed methods for breakpoint resolution: 

// Request-phase modify
mockServerClient.modifyBreakpoint(pauseId, modifiedRequest);

// Response-phase modify
mockServerClient.modifyBreakpointResponse(pauseId, modifiedResponse);
