MockServer can drive outbound API traffic on demand using Load Scenarios — a declarative, bounded load generator built into the server. A load scenario is an ordered list of templated request steps fired at a target concurrency described by a load profile, with per-iteration variable data. Results feed both the metrics histograms and the SLO sample store so a load run can be asserted with PUT /mockserver/verifySLO.

This makes MockServer useful for resilience verification: inject a chaos fault (see Chaos Testing), apply load, then assert that your SLOs held — all from a single control plane, with no external load tool required.

Off by default. The endpoint returns 403 Forbidden until loadGenerationEnabled=true is set. Hard caps on virtual users, in-flight requests, RPS, duration, and step count prevent the feature from self-DoS-ing the server.

 

Quickstart

Enable load generation, then start a 10-VU constant load scenario against a target for 30 seconds:

# 1. Enable load generation (set once at startup or via the config endpoint)
docker run -e MOCKSERVER_LOAD_GENERATION_ENABLED=true mockserver/mockserver

# 2. Start a scenario
curl -s -X PUT http://localhost:1080/mockserver/loadScenario \
  -H "Content-Type: application/json" \
  -d '{
    "name": "checkout-load",
    "templateType": "VELOCITY",
    "profile": {
      "type": "CONSTANT",
      "vus": 10,
      "durationMillis": 30000
    },
    "steps": [
      {
        "request": {
          "method": "GET",
          "path": "/api/orders/$iteration.index",
          "headers": { "Host": ["orders.svc:8080"] },
          "socketAddress": { "host": "orders.svc", "port": 8080, "scheme": "HTTP" }
        }
      }
    ]
  }'

# 3. Poll progress
curl -s http://localhost:1080/mockserver/loadScenario

# 4. Stop early if needed
curl -s -X DELETE http://localhost:1080/mockserver/loadScenario
 

Control-Plane API

All three verbs are control-plane endpoints — they respect control-plane authentication (mTLS / JWT) when configured.

Verb Path Behaviour
PUT /mockserver/loadScenario Start a new scenario. Returns 403 when loadGenerationEnabled=false; 400 with a JSON error when the scenario fails validation or a cap is exceeded; 200 {status:"started", ...} otherwise. Starting a new scenario while one is running replaces the previous one.
GET /mockserver/loadScenario Return current scenario status. Fields: state (running / completed / stopped / none), elapsedMillis, currentVus, requestsSent, succeeded, failed, p50Millis, p95Millis, p99Millis, runId, startedAt, endedAt.
DELETE /mockserver/loadScenario Stop the running scenario. Idempotent — no-op when nothing is running.
 

Load Scenario Model

A load scenario is a JSON object with the following top-level fields:

Field Type Required Description
name string Yes Display name for the scenario (appears in status and logs).
steps array Yes Ordered list of request steps (see Steps below). Maximum 50 steps.
profile object Yes The ramp profile controlling virtual users and duration (see Load Profile below).
templateType string No Template engine for rendering step fields: VELOCITY (default) or MUSTACHE. JAVASCRIPT is not supported for load steps and is rejected with a 400.
maxRequests integer No Stop the scenario once this many requests have been dispatched, even if the duration has not elapsed. Useful for budget-bounded runs.
 

Steps

Each step defines a request to fire and an optional pause after it:

Field Type Description
request HttpRequest The request to fire. Reuses the same HttpRequest model as expectations. Template expressions live in the path and body fields. Set socketAddress to point at the target service.
thinkTime Delay Optional pause between this step and the next (a { "timeUnit": "MILLISECONDS", "value": 100 } object). Implemented with a non-blocking scheduler — no thread is blocked during think time.
 

Load Profile

The profile controls how many virtual users (VUs) are active over time:

Field Type Description
type string CONSTANT — a fixed number of VUs for the whole duration. LINEAR — ramp VUs linearly from startVus to endVus.
durationMillis integer How long to run (milliseconds). Maximum 3 600 000 ms (1 hour).
vus integer CONSTANT only — number of concurrent virtual users. Maximum 50.
startVus integer LINEAR only — virtual users at the start of the ramp.
endVus integer LINEAR only — virtual users at the end of the ramp. Maximum 50.
iterationPacingMillis integer Optional minimum time between the start of successive iterations for a given VU. Limits the iteration rate per VU independently of step think times.
 

Per-Iteration Template Variables

Each iteration gets a fresh iteration context injected alongside the standard request variable. Use it to vary data across iterations and virtual users without external data files:

Variable Meaning Velocity Mustache
index Global iteration index across all VUs (0-based) $iteration.index {{iteration.index}}
vuId ID of the virtual user running this iteration (0-based) $iteration.vuId {{iteration.vuId}}
vuIteration Iteration count within this specific VU (0-based) $iteration.vuIteration {{iteration.vuIteration}}
elapsedMillis Milliseconds since the scenario started $iteration.elapsedMillis {{iteration.elapsedMillis}}
count Total requests dispatched so far (across all VUs) $iteration.count {{iteration.count}}

What fields are rendered. In v1, only the request path and body fields are rendered through the template engine. These are the most commonly templated fields — use them to vary the URL path or the request payload across iterations.

 

Examples

CONSTANT profile — templated path

Ten VUs, 30 seconds, each iteration fetches a different order ID derived from the global iteration index:

curl -s -X PUT http://localhost:1080/mockserver/loadScenario \
  -H "Content-Type: application/json" \
  -d '{
    "name": "checkout-load",
    "templateType": "VELOCITY",
    "maxRequests": 5000,
    "profile": {
      "type": "CONSTANT",
      "vus": 10,
      "durationMillis": 30000,
      "iterationPacingMillis": 50
    },
    "steps": [
      {
        "request": {
          "method": "GET",
          "path": "/api/orders/$iteration.index",
          "headers": { "Host": ["orders.svc:8080"] },
          "socketAddress": { "host": "orders.svc", "port": 8080, "scheme": "HTTP" }
        },
        "thinkTime": { "timeUnit": "MILLISECONDS", "value": 20 }
      }
    ]
  }'

LINEAR ramp

Ramp from 1 to 25 VUs over 60 seconds — useful for finding the concurrency at which a service starts to degrade:

curl -s -X PUT http://localhost:1080/mockserver/loadScenario \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ramp-to-25",
    "profile": {
      "type": "LINEAR",
      "startVus": 1,
      "endVus": 25,
      "durationMillis": 60000
    },
    "steps": [
      {
        "request": {
          "method": "GET",
          "path": "/health",
          "socketAddress": { "host": "target.svc", "port": 8080, "scheme": "HTTP" }
        }
      }
    ]
  }'

Multi-step scenario with per-VU data variation

A two-step user journey: create a basket, then add an item. The basket ID is derived from the VU, and the product ID from the iteration within that VU:

curl -s -X PUT http://localhost:1080/mockserver/loadScenario \
  -H "Content-Type: application/json" \
  -d '{
    "name": "basket-journey",
    "templateType": "VELOCITY",
    "profile": {
      "type": "CONSTANT",
      "vus": 5,
      "durationMillis": 60000
    },
    "steps": [
      {
        "request": {
          "method": "POST",
          "path": "/baskets",
          "body": { "type": "STRING", "string": "{\"customerId\": $iteration.vuId}" },
          "socketAddress": { "host": "shop.svc", "port": 8080, "scheme": "HTTP" }
        },
        "thinkTime": { "timeUnit": "MILLISECONDS", "value": 50 }
      },
      {
        "request": {
          "method": "POST",
          "path": "/baskets/$iteration.vuId/items",
          "body": { "type": "STRING", "string": "{\"productId\": $iteration.vuIteration}" },
          "socketAddress": { "host": "shop.svc", "port": 8080, "scheme": "HTTP" }
        }
      }
    ]
  }'
 

Safety Caps

Load generation is bounded by hard caps to prevent the feature from self-DoS-ing the server. Requests that exceed a cap are rejected with 400 Bad Request and a JSON error message.

Control Default cap Enforced at
Feature flag (loadGenerationEnabled) false (disabled) PUT returns 403 when off
Maximum virtual users 50 Scenario validation (rejected at PUT)
Maximum in-flight requests 200 Live in-flight semaphore at dispatch
Maximum requests per second 500 Live token bucket at dispatch
Maximum duration 3 600 000 ms (1 hour) Scenario validation (rejected at PUT)
Maximum steps per scenario 50 Scenario validation (rejected at PUT)
 

Asserting SLOs over Load Results

Every completed load-scenario request is recorded into MockServer's SLO sample store (when sloTrackingEnabled=true), so you can drive load and then assert that latency and error-rate objectives held — all from the MockServer control plane.

# 1. Enable both features
docker run \
  -e MOCKSERVER_LOAD_GENERATION_ENABLED=true \
  -e MOCKSERVER_SLO_TRACKING_ENABLED=true \
  mockserver/mockserver

# 2. Run a load scenario (e.g. CONSTANT 10 VUs for 30 s)
curl -s -X PUT http://localhost:1080/mockserver/loadScenario \
  -H "Content-Type: application/json" \
  -d '{
    "name": "slo-validation-run",
    "profile": { "type": "CONSTANT", "vus": 10, "durationMillis": 30000 },
    "steps": [
      {
        "request": {
          "method": "GET",
          "path": "/api/health",
          "socketAddress": { "host": "target.svc", "port": 8080, "scheme": "HTTP" }
        }
      }
    ]
  }'

# 3. Wait for the scenario to complete (poll GET until state = "completed")

# 4. Assert SLOs held during the run
# Returns 200 (PASS) or 406 (FAIL)
curl -s -w "\n%{http_code}" -X PUT http://localhost:1080/mockserver/verifySLO \
  -H "Content-Type: application/json" \
  -d '{
    "name": "checkout-slo",
    "window": { "type": "LOOKBACK", "lookbackMillis": 60000 },
    "minimumSampleCount": 50,
    "upstreamHosts": ["target.svc"],
    "objectives": [
      { "sli": "LATENCY_P95", "comparator": "LESS_THAN", "threshold": 200.0 },
      { "sli": "ERROR_RATE",  "comparator": "LESS_THAN_OR_EQUAL", "threshold": 0.01 }
    ]
  }'

See SLO Resilience Verdicts for the full verifySLO reference, including window types, all SLI options, and the response body schema.

Scope note: the SLO sample store records load-scenario traffic and real proxied traffic under the same FORWARD scope, keyed by target host. When asserting SLOs over a load run, use a narrow window that covers only the load period to avoid mixing in unrelated traffic.

 

Configuration

Property Environment variable Default Description
mockserver.loadGenerationEnabled MOCKSERVER_LOAD_GENERATION_ENABLED false Master switch. Must be true for PUT /mockserver/loadScenario to succeed. Set at startup to avoid a restart.
